Intel TSX-NI: Hardware Lock Elision Performance

Variant 15: MD = min(MT)*(MT+MX) - MT*MX; D = (MT+MX)*B - (MT-MX)*E

Synchronization: ReentrantLock (java.util.concurrent.locks.Lock) with fine-grained per-element locking. When TSX-NI is enabled, the CPU can speculatively execute lock-protected regions without actually acquiring the lock (Hardware Lock Elision).

Conclusions

When Intel TSX-NI (Transactional Synchronization Extensions) is enabled, the CPU uses Hardware Lock Elision (HLE) to speculatively execute critical sections protected by ReentrantLock without actually acquiring the lock. Since each thread writes to non-overlapping regions of the result arrays (MD, D), there are no actual data conflicts, and the speculative execution succeeds. This eliminates lock contention overhead.

When TSX-NI is disabled, every lock acquisition is a real atomic compare-and-swap operation, introducing serialization overhead even though threads access different memory locations. The fine-grained locking pattern (lock per element write) amplifies this effect, as the lock is acquired and released N*N times for MD and N times for D.

The speedup from TSX-NI is expected to increase with larger N, as the total number of lock operations grows quadratically with matrix size.

TSX-NI Control

Enable TSX-NI:
reg add "HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\Kernel" /v DisableTsx /t REG_DWORD /d 0 /f

Disable TSX-NI:
reg add "HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\Kernel" /v DisableTsx /t REG_DWORD /d 1 /f

Reboot required after each change.
